# ChaCha20-Poly1305 RFC 8439 — Poly1305 MAC

<poly1305_algorithm>
## The Poly1305 Algorithm (§2.5)

One-time authenticator. Takes a 32-byte one-time key and a message, produces a 16-byte (128-bit) tag.

### Key Structure
The 256-bit key is split into two 128-bit halves:
- **r** (first 16 bytes) — the "secret multiplier", clamped before use
- **s** (last 16 bytes) — the "secret addend", added at the end

The pair (r, s) **MUST be unique and unpredictable per invocation**. r MAY be constant across invocations, but s MUST NOT be reused.

### Clamping r (§2.5)
Before use, r is modified to ensure certain algebraic properties:
```
r[3], r[7], r[11], r[15]  &= 0x0f    // top 4 bits cleared (< 16)
r[4], r[8], r[12]         &= 0xfc    // bottom 2 bits cleared (÷4)
```

Or equivalently as a single mask: `r &= 0x0ffffffc0ffffffc0ffffffc0fffffff`

### Algorithm

Prime: **P = 2^130 − 5** (= 0x3fffffffffffffffffffffffffffffffb)

```
poly1305_mac(msg, key):
    r = le_bytes_to_num(key[0..15])
    clamp(r)
    s = le_bytes_to_num(key[16..31])
    accumulator = 0

    for each 16-byte block of msg:
        n = le_bytes_to_num(block) + 2^(8 * block_length)   // append 0x01 byte
        accumulator += n
        accumulator = (accumulator * r) mod P

    accumulator += s
    return num_to_16_le_bytes(accumulator)    // take lowest 128 bits
```

### Block Processing Detail
- Read block as little-endian integer
- **Add a 1 bit beyond the block length:** full 16-byte block → add 2^128. Last block of k bytes → add 2^(8k). This prevents zero-padding attacks.
- Add to accumulator, multiply by r, reduce mod P
- The `+ s` at the end is NOT reduced mod P — raw 128-bit addition, then take low 128 bits

### Security Properties
- **One-time:** Key MUST NOT be reused. Reusing r with different s is also unsafe.
- Forgery probability: 1 − (n/(2^102)) for a 16n-byte message, even after 2^64 legitimate messages
- SUF-CMA (strong unforgeability against chosen-message attacks)
- **NOT suitable as a PRF** — biased output, key reuse prohibited

## Generating the Poly1305 Key Using ChaCha20 (§2.6)

In the AEAD construction, the Poly1305 one-time key is derived from the ChaCha20 encryption key:

```
poly1305_key_gen(key, nonce):
    counter = 0
    block = chacha20_block(key, counter, nonce)
    return block[0..31]     // first 256 bits only; discard rest
```

- Uses block counter **0** (the AEAD encryption itself starts at counter **1**)
- First 128 bits → r (clamped), next 128 bits → s
- Remaining 256 bits of the 512-bit block output are discarded
- The nonce MUST be unique per invocation, MUST NOT be randomly generated — counters or LFSRs are acceptable
